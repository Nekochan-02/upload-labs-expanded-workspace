# Handoff (第2次実装工程への引き継ぎ資料)

第1次解析フェーズの完了に伴う、第2次実装工程（Modの試作・デバッグ）を担当する開発エンジニア向けの引き継ぎドキュメントです。

---

## 1. 現状のステータス要約

* **開発環境の整備**:
  * 専用ワークスペース (`upload-labs-expanded-workspace/`) を構築。
  * バニラゲーム本体のコードは `vanilla-reference/` ディレクトリに GDScript 2.0 形式で完全に復元済み。
  * `.gitignore` によりバニラコードや生成ログが絶対にGit管理下に混入しない隔離環境を確立。
* **解析の進捗**:
  * ノード総数上限（`MAX_WINDOW = 500`）および配置可能領域（ハードコードされた `10000` サイズの境界）の制御コード経路を100%特定。
  * `Godot Mod Loader v7` のアドオン構造を確認し、最小限の競合でパッチを適用するための `Script Extension` 方式の適用を決定。

## 2. 次工程への具体的な推奨パッチポイント

### 2.1 ノード上限の緩和パッチ
定数 `MAX_WINDOW` は直接変更できないため、以下の関数をオーバーライドして、Mod設定から読み込んだカスタム最大値を返すように変更します。
* **対象ファイル**: `scripts/utils.gd`
* **対象関数**: `can_add_window(window: String) -> bool`
* **修正方針**: `MAX_WINDOW` を直接比較している代わりに、Modで設定したカスタム上限値を返すヘルパー関数を作成して適用する。
* **注意**: `desktop.gd` と `window_dragger.gd` の中の `MAX_WINDOW` 判定処理も `Script Extension` で拡張し、定数の代わりにMod側の最大ノード数を参照するように差し替える。

### 2.2 配置領域の拡張パッチ
キャンバスサイズとして、`10000` がハードコードされている部分を、拡張サイズ（例: `20000`）に一括置換または変数参照に変更します。
1. **`main_2d.gd` (L6)**: `screen_size[0] = 20000` に上書きしてカメラ制限の限界を拡張。
2. **`windows/window_container.gd` (L156)**: `get_position_snapped` 内の `to.clamp(Vector2.ZERO, (Vector2.ONE * 20000) - size)` でウィンドウドラッグ時のクランプ範囲を広げる。
3. **`connector_point.gd` (L27)**: `pos = to.clampf(0, 20000)` に上書き。
4. **`desktop.gd` (L195)**: `paste` のクランプ範囲を `20000` に拡張。
5. **`lines.gd`**: グリッド描画用のループカウンタ（`build_lines` や `build_circles` などのループ上限）を `20000 / 50` 相当に拡張する。

## 3. 最優先で実行すべきネクストアクション

1. **Mod初期パッケージの作成**:
   * `mod/source` に `manifest.json` と `mod_main.gd` を作成。
   * エントリーポイント `mod_main.gd` の `_init()` にて `ModLoaderMod.install_script_extension()` を呼び出し、上記パッチの拡張スクリプト群をバインドする。
2. **2倍サイズ（20000）のモックパッチテスト**:
   * 最初から極端なサイズや無制限化を行わず、バニラの2倍（ノード数1000、サイズ20000）を設定して、配置、カメラ移動、グリッドの描画がシームレスに行えるか検証する。
3. **パフォーマンスフットプリントの計測**:
   * グリッドを `MultiMesh` で描画する際、インスタンス数が16万個に増えたとき（2倍サイズ）の起動時プチフリーズの有無を確認する。
